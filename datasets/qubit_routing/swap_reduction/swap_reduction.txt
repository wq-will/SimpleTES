## 1) Formal problem statement

### Inputs

You are given:

1. A **quantum circuit** (C) over logical qubits ($L={0,\dots,n-1}$).

2. A **hardware topology** ($G=(P,E)$), with physical qubits ($P={0,\dots,m-1}$) and undirected edges ($E\subseteq \{(p,q)\mid p,q\in P\}$), where a 2-qubit gate can be executed natively only on an edge.

### Output

You must produce a routed physical circuit (C') that is valid under (G). Formally, routing consists of:

* an initial injective mapping ($\pi_0: L \to P$),
* a sequence of SWAP layers (${S_t \subseteq E}_{t=0}^{T-1}$) that update the mapping over time,
* a gate schedule that respects the circuit dependencies and ensures that each 2-qubit gate ($(\ell_i,\ell_j)$) is executed only when (${\pi_t(\ell_i),\pi_t(\ell_j)}\in E$).

## Goal

The evaluator computes a score over a benchmark suite of different quantum circuits and hardware topologies. Your optimization objective:

* **Maximize `combined_score`**.

For each benchmark case `i`, let:

* `w_i` = the case weight from suite metadata,
* `orig_i` = inserted CNOT count (1 SWAP = 3 CNOTs) by the baseline routing strategy
* `add_i` = inserted CNOT count by your proposed strategy.
    * if routing succeeds: `add_i = 3 * the routed circuit's inserted SWAP count`
    * if routing fails / is invalid: `add_i = orig_i` (that case gets no credit)

The evaluator uses:
* total score: `combined_score = sum_i w_i * (orig_i - add_i)`

So higher is better. In practice, this means:

* reducing inserted SWAPs is the main driver of score,
* failures are very expensive because they zero out that case's improvement,

The evaluator also reports `depth`, runtime, validity, and other aggregate metrics. Those are useful signals when reasoning about tradeoffs, but `combined_score` is the actual optimization target.

If your policy fails to compile, panics, times out, or produces invalid routing on any benchmark instance, it receives a large penalty / fails.

impl<'a> FrontLayerView<'a> {
    pub(crate) fn new(
        circuit: CircuitDagView<'a>,
        layout: LayoutView<'a>,
        node_ids: &'a [usize],
        physical_pairs: &'a [[usize; 2]],
    ) -> Self {
        Self {
            circuit,
            layout,
            node_ids,
            physical_pairs,
        }
    }

    pub fn len(&self) -> usize {
        self.node_ids.len()
    }

    pub fn is_empty(&self) -> bool {
        self.node_ids.is_empty()
    }

    pub fn node_ids(&self) -> &[usize] {
        self.node_ids
    }

    pub fn logical_pair(&self, node_id: usize) -> Option<(usize, usize)> {
        self.circuit.node(node_id).two_qubit_pair()
    }

    pub fn physical_pair(&self, node_id: usize) -> Option<(usize, usize)> {
        let idx = self.node_ids.iter().position(|&id| id == node_id)?;
        let pair = self.physical_pairs[idx];
        Some((pair[0], pair[1]))
    }

    pub fn active_logical_qubits(&self) -> Vec<usize> {
        let mut out = Vec::new();
        let mut seen = HashSet::new();
        for &node_id in self.node_ids {
            if let Some((a, b)) = self.logical_pair(node_id) {
                if seen.insert(a) {
                    out.push(a);
                }
                if seen.insert(b) {
                    out.push(b);
                }
            }
        }
        out
    }

    pub fn active_physical_qubits(&self) -> Vec<usize> {
        let mut out = Vec::new();
        for pair in self.physical_pairs {
            out.push(pair[0]);
            out.push(pair[1]);
        }
        out
    }

    pub fn physical_pairs(&self) -> &'a [[usize; 2]] {
        self.physical_pairs
    }
}

    pub fn shuffle_slice<T>(&mut self, values: &mut [T]) {
        if values.len() <= 1 {
            return;
        }
        for i in (1..values.len()).rev() {
            let j = self.rng.random_range(0..=i);
            values.swap(i, j);
        }
    }

## 4) Baseline: LightSABRE (what you’re starting from)

Your initial `candidate.rs` is an implementation of Qiskit LightSABRE, You can mutate it to achieve a better algorithm.

### Core heuristic skeleton

It is a greedy heuristic-search router that maintains a **front layer** of currently executable two-qubit gates in the dependency DAG, enumerates **candidate SWAPs touching qubits in the front layer**, and chooses the best SWAP by a cost function based primarily on **graph distances** between mapped qubits for the front layer plus a **lookahead set**, with an optional **decay** factor to discourage repeatedly swapping the same qubits.

There are several techniques to improve runtime and quality:

* efficient scoring (relative/delta evaluation rather than full recomputation),
* multiple trials,
* “release valve” behavior to guarantee progress on hard cases,
* broader support obligations such as disjoint coupling graphs and control flow.
